# GENERATOR_FEASIBILITY.md

Maintenance Parity Closure Program — Phase 2.

This audit classifies each important metadata item by whether it can be generated and what proof-of-concept path would make it repeatable.

## Classification legend

- `GENERATED_FROM_GO`: generated directly from upstream Go source text or generated Go files.
- `GENERATED_FROM_FIXTURES`: generated from Go-side fixture programs that execute/reflection-scan upstream behavior and emit JSON/TS.
- `GENERATED_FROM_AST`: generated by parsing Go AST/packages and extracting declarations/signatures.
- `GENERATED_FROM_REFLECTION_SCAN`: generated by a small Go program importing `references/expr` and using `reflect` on actual values/types.
- `MANUAL_ONLY`: cannot be fully generated in a useful way; must remain a designed TS adapter with tests.

## Proof-of-concept designs

### PoC A — Generate `FUNC_TYPES`

Input:

```text
references/expr/vm/func_types[generated].go
```

Parsing rule:

```text
N: new(func(<params>) <returns>)
```

Output:

```ts
// src/checker/func_types.generated.ts
export const FUNC_TYPES = [
  ft([], []),
  ft([], [_dur]),
  ft([_arrAny, _string], [_string]),
];
export const FUNC_TYPES_ARITY = FUNC_TYPES.map((f) => f.in.length);
```

Feasibility: high. The Go generated file is regular and already index-labeled. This removes the most dangerous manual table.

### PoC B — Generate standard reflect descriptors

Input Go program:

```go
package main

import (
  "encoding/json"
  "reflect"
  "time"
)

func main() {
  values := map[string]reflect.Type{
    "time.Time": reflect.TypeOf(time.Time{}),
    "time.Duration": reflect.TypeOf(time.Duration(0)),
    "*time.Location": reflect.TypeOf(new(time.Location)),
  }
  // emit kind, name, elem/key, methods with in/out type strings
  _ = json.NewEncoder(os.Stdout).Encode(values)
}
```

Output:

```ts
// src/checker/std_types.generated.ts
export const timeType = makeStruct("time.Time", [...methods]);
export const durationType = makeNamedInt64("time.Duration", [...methods]);
export const locationType = makePtr("*time.Location", ...);
```

Feasibility: high for stdlib method metadata. It should replace duplicated `locationType` / `locationTypeRef` definitions.

### PoC C — Generate visible struct field metadata

Input Go scanner:

```go
t := reflect.TypeOf(ExportedEnv{})
fields := reflect.VisibleFields(t)
```

Output JSON:

```json
{
  "name": "ExportedEnv",
  "fields": [
    {"name":"ExportedEmbedded","type":"ExportedEmbedded","index":[0],"pkgPath":""},
    {"name":"Str","type":"string","index":[0,0],"pkgPath":""}
  ],
  "hidden": ["unexportedEmbedded", "str"]
}
```

Output TS:

```ts
markStruct(env, "ExportedEnv", generatedFields.ExportedEnv.fields);
```

Feasibility: high for test/env metadata. This would turn visibility adapters from manual to reflection-generated fixtures.

### PoC D — Generate valuer method list

Input:

```text
references/expr/patcher/value/value.go
```

Parse interface names/methods or type switch cases in priority order:

```go
case AsAny:
case AsBool:
case AsInt:
...
```

Output:

```ts
export const VALUER_METHODS = ["AsAny", "AsBool", "AsInt", ...] as const;
```

Feasibility: high for method list/order. The TS `ValuePatcher` strategy remains manual.

### PoC E — Generate time layout fixtures

Input Go program loops layouts/timezones/instants:

```go
for _, layout := range layouts {
  fmt.Println(t.In(loc).Format(layout))
}
```

Output JSON fixtures consumed by TS tests.

Feasibility: high for behavior coverage, not code generation. This is the right approach because Go time layout code is complex and not useful to transpile directly.

## Items that should remain manual

1. `Type` class and descriptor schema: TS adapter replacing reflect.
2. `STRUCT_TYPE` symbol and `markStruct()` carrier mechanism: JS-specific metadata design.
3. Runtime `GoTime`, `GoDuration`, `GoLocation` classes: JS host adapter.
4. Brand-aware runtime wrapper design: JS-specific named-type representation.
5. Checker/patcher strategies where Go uses reflection dynamically: source-port logic plus generated metadata is safer than generated TS code.

## Phase 2 conclusion

The highest-risk manual data can be mostly generated:

- `FUNC_TYPES` → `GENERATED_FROM_GO`
- standard time/location descriptors → `GENERATED_FROM_REFLECTION_SCAN`
- visible struct metadata → `GENERATED_FROM_REFLECTION_SCAN`
- valuer method list → `GENERATED_FROM_GO`
- named-type test env wrappers → `GENERATED_FROM_FIXTURES`

But the core reflection replacement architecture cannot be generated away. It can only be fed by generated metadata and guarded by generated fixtures.
