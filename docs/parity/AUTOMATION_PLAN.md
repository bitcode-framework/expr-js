# AUTOMATION_PLAN.md

Maintenance Parity Closure Program — Phase 4.

Concrete plan to remove or reduce manual knowledge so important metadata has a single source of truth.

## Principles

1. Do not generate behavior-heavy TS code when source-porting is clearer.
2. Generate data and fixtures that Go can authoritatively provide.
3. Keep TS adapter architecture explicit and tested.
4. Every generated file must include source file, upstream version, command, and checksum/header.
5. Generated outputs should be committed only after full verification.

## Target architecture

```text
references/expr/**.go
        ↓
metadata generators / Go reflection scanners / fixture generators
        ↓
src/**.generated.ts + tests/**/fixtures.generated.json
        ↓
existing checker/compiler/vm/runtime consume generated data
```

## Plan by area

### 1. `FUNC_TYPES`

Current:

```ts
// src/checker/info.ts
const FUNC_TYPES = [...]
export const FUNC_TYPES_ARITY = []
```

Target:

```text
references/expr/vm/func_types[generated].go
  ↓ scripts/gen-func-types.mjs
src/checker/func_types.generated.ts
  ↓ imported by src/checker/info.ts and src/vm/vm.ts
```

Generated output:

```ts
// Code generated by scripts/gen-func-types.mjs from references/expr/vm/func_types[generated].go. DO NOT EDIT.
export const FUNC_TYPES = [...];
export const FUNC_TYPES_ARITY = FUNC_TYPES.map((entry) => entry.in.length);
```

Steps:

1. Write parser for `new(func(...) ...)` entries.
2. Map Go type strings to TypeDescriptor constructors.
3. Generate table and arity.
4. Add snapshot test comparing generated index count and selected indices: 32, 76, variadic exclusions.
5. Remove inline table from `info.ts`.

Status target: `GENERATED_FROM_GO`.

### 2. Standard reflect descriptors

Current:

```ts
// src/checker/nature/type.ts
export const timeType = new Type(...)
// manual methods

// src/builtin/utils.ts
export const locationType = ...

// src/checker/nature/nature.ts
const locationTypeRef = ... // duplicate
```

Target:

```text
references/expr + Go stdlib reflect scan
  ↓ parity/metagen/stdtypes/main.go
src/checker/nature/std_types.generated.ts
```

Generated output:

```ts
export const timeType = makeStructType("time.Time", generatedMethods);
export const durationType = makeNamedInt64Type("time.Duration", generatedMethods);
export const locationType = makePtrType("*time.Location", generatedMethods);
```

Steps:

1. Create Go scanner that emits JSON for selected standard types.
2. Create TS generator consuming JSON and emitting TypeDescriptor construction.
3. Replace duplicate `locationTypeRef` with import from generated std types.
4. Add tests for method availability: `Format`, `String`, `Year`, `Hours`, `timezone(...).String()`.

Status target: `GENERATED_FROM_REFLECTION_SCAN`.

### 3. Visible struct metadata

Current:

```ts
markStruct(env, "ExportedEnv", { ...manual visible fields... })
```

Target:

```text
references/expr/test/issues/* Go env structs
  ↓ parity/metagen/visible-fields/main.go using reflect.VisibleFields
parity/metadata/visible_fields.generated.json
  ↓ scripts/gen-visible-structs.mjs
src/checker/fixtures/visible_structs.generated.ts OR tests metadata fixture
```

Generated output example:

```ts
export const ExportedEnv = makeStructMeta({
  name: "ExportedEnv",
  fields: [...],
  methods: [...],
});
```

Steps:

1. Start with issue 844 env types.
2. Emit field name/type/index/anonymous/PkgPath from Go.
3. Generate TS `StructMeta` constants.
4. Update tests to call `applyStructMeta(env, Generated.ExportedEnv)` instead of hand-declaring fields.
5. Expand scanner to mock env and other issue envs.

Status target: `GENERATED_FROM_REFLECTION_SCAN` for test/env metadata.

### 4. Named primitive metadata / branded values

Current:

```ts
brand(1n, "ModeEnum")
```

Target:

```text
Go fixture generator sees value type ModeEnum
  ↓ fixture JSON includes {typeName:"ModeEnum", underlying:"int"}
TS fixture loader automatically wraps with brand(value, typeName)
```

Generated fixture value example:

```json
{"k":"int","v":"1","typeName":"ModeEnum"}
```

Steps:

1. Extend Go fixture normalizer to include named type metadata for primitives.
2. Extend TS fixture loader to call `brand()` when `typeName` exists.
3. Remove issue-test-only manual `brand()` where fixture-based.
4. Keep `branded.ts` runtime wrapper as manual adapter.

Status target: `GENERATED_FROM_FIXTURES` for data; `MANUAL_ONLY` for wrapper design.

### 5. Builtin registry and validator drift detection

Current:

```ts
// src/builtin/builtin.ts
Builtins.push(new Func({ Name, Types, Validate, ... }))
```

Target:

```text
references/expr/builtin/builtin.go
  ↓ Go AST metadata extractor
parity/metadata/builtins.generated.json
  ↓ tests compare TS Builtins metadata snapshot
```

Do not generate validator implementation initially. Generate a diff detector.

Steps:

1. Parse Go AST for builtin `Name`, `Predicate`, `Types` expression strings, and `Validate` presence.
2. Emit JSON metadata snapshot.
3. Add TS test that compares TS `Builtins` names/order/predicate/arity when representable.
4. Use generated fixture tests for validator behavior.

Status target: `GENERATED_FROM_AST` for drift detection; executable validators remain manual source ports.

### 6. Valuer method list

Current:

```ts
const VALUER_METHODS = ["AsAny", "AsBool", ...]
```

Target:

```text
references/expr/patcher/value/value.go
  ↓ scripts/gen-valuer-methods.mjs or Go AST scanner
src/patcher/value/valuer_methods.generated.ts
```

Generated output:

```ts
export const VALUER_METHODS = ["AsAny", "AsBool", ...] as const;
```

Steps:

1. Parse type-switch order or interface declarations in `value.go`.
2. Emit generated const array.
3. Import in `value.ts`.
4. Add test that generated order matches Go source scan.

Status target: `GENERATED_FROM_GO`.

### 7. Time layout behavior fixtures

Current:

Manual `gotime.ts` parser/formatter plus limited parity tests.

Target:

```text
Go time fixture generator
  ↓ parity/fixtures/time_layout.generated.json
TS test validates GoTime.Format/parse behavior
```

Steps:

1. Generate matrix of layouts, instants, locations.
2. Include edge layouts: zone offsets, month/day names, zero-padding, fractional seconds.
3. Test TS `GoTime.Format()` and date parsing against expected Go output.
4. Keep implementation manual but fixture-driven.

Status target: `GENERATED_FROM_FIXTURES` for behavior coverage.

### 8. TypeDescriptor schema guard

Current:

Manual `Type` / `FieldDescriptor` schema.

Target:

Generated conformance fixtures, not generated schema.

Steps:

1. Reflection scanner emits descriptors for representative structs/functions/maps/slices/pointers.
2. TS tests assert generated metadata can be represented losslessly by `Type`.
3. When scanner emits unsupported metadata, test fails and schema must be extended.

Status target: `MANUAL_ONLY` design with generated conformance tests.

```json
{
  "scripts": {
    "gen:metadata": "npm run gen:func-types && npm run gen:std-types && npm run gen:visible-structs && npm run gen:valuer-methods && npm run gen:time-fixtures",
    "check:metadata": "npm run gen:metadata && git diff --exit-code -- src parity tests"
  }
}
```

Actual script names can differ; this is the automation target, not an implemented change.

## Priority order

1. `FUNC_TYPES` generator — removes highest-risk bytecode-visible manual table.
2. `std_types.generated.ts` — removes duplicated location/time/duration descriptor metadata.
3. visible struct metadata generator — removes manual visibility knowledge in strict struct tests.
4. named-type fixture tags — removes manual `brand()` usage in parity fixtures.
5. valuer method generator — removes ordered interface method list.
6. builtin metadata drift detector — catches upstream builtin changes.
7. time layout fixture generator — expands behavior proof.
8. TypeDescriptor conformance scanner — prevents future reflect metadata blind spots.

## Definition of success

Maintenance metadata can be considered mostly automated when:

- `FUNC_TYPES` is generated from Go.
- Standard time/duration/location descriptors are generated from reflection scan.
- Struct visible-field metadata in tests is generated from Go reflection.
- Named primitive fixture values carry generated type names.
- Valuer method list is generated from Go source.
- `check:metadata` fails if generated outputs are stale.

## Phase 4 conclusion

Manual knowledge cannot be eliminated completely because TypeScript still needs an explicit reflection adapter. But the dangerous manual data can be moved behind generators. The realistic target is `MOSTLY_AUTOMATED` metadata maintenance, not `FULLY_AUTOMATED` whole-port maintenance.
